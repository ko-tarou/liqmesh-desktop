# LiqMesh Desktop 全体計画

LiqMesh の Windows デスクトップクライアント。オンライン(Supabase)断時に
Android/iOS と **BLE central 専用**で直接 P2P チャットする。
正典は [`docs/BLE_CONTRACT.md`](../../docs/BLE_CONTRACT.md)（改変禁止）。

技術スタック: Tauri 2 + React 19 + TypeScript + Vite 7（フロント）/ Rust（BLE は src-tauri 側）。

## Phase 一覧

| Phase | 内容 | 状態 |
|---|---|---|
| Phase 1 | Tauri 雛形 + BLE_CONTRACT.md 正典保存 + 計画ファイル | 本 PR で完了 |
| Phase 2 | BLE central 実装（btleplug, strict TDD） | 未着手 |

## Phase 1（本フェーズ）

スコープ: 動く雛形と 3 セッション共通の正典を main に乗せる。重い依存
（llama.cpp / btleplug）はまだ入れない。

- Tauri React-TS 雛形（src-tauri Rust + Vite/React/TS フロント）
- `.gitignore` 整備（node_modules, dist, target, src-tauri/target, .DS_Store）
- `docs/BLE_CONTRACT.md` を正典テキストで保存（一字一句固定）
- ビルド確認: `npm run build`（フロント）/ `cargo check`（Rust）
- 計画ファイル 2 本

PR: 不要（空 repo の初期 scaffold のため main へ直接コミット→push）。

## Phase 2（次フェーズ・strict TDD）

スコープ: `docs/BLE_CONTRACT.md` に厳密準拠した BLE central 機能。

実行順序（依存順・各 PR は 200 行以内 / 50 行/コミット目安・全て main 向け）:

| # | PR | 内容 | 優先度 | 依存 |
|---|---|---|---|---|
| 1 | チャンク再構成 | `[msgId:4][seq:1][total:1][payload]` の分割/再構成。MTU 247→23 フォールバック計算。先にテストを書く | 高 | — |
| 2 | wire JSON 型 | hello/msg/reaction/delete/read のシリアライズ。未知 type は無視（前方互換）。テスト先行 | 高 | 1 |
| 3 | BLE central スキャン/接続 | btleplug 導入。Service UUID `B1E5C0DE-...0001` でスキャン→接続→RX(0003) 購読→TX(0002) 書込 | 高 | 1,2 |
| 4 | セッション層 | 接続後 hello 双方向必須。senderId を接続単位に束縛（TOFU）。永続セッション保持 | 中 | 3 |
| 5 | UI 連携 | Tauri command/event でフロントへ受信メッセージを流す。送信フロー | 中 | 4 |

TDD 方針: 各 PR は失敗するテストを先に書き、最小実装で通す。チャンク/wire は
純ロジックなのでユニットテストで網羅。BLE I/O 層はトレイト抽象でモック可能にする。

## Core 原則

- Simplicity First / Minimal Impact / No Laziness
- Desktop は常に central（advertise 不可）— contract のとおり
- v1 は直接リンクのみ（マルチホップ中継は v2）
